# Claude Code 작업 지침

이 파일은 Claude Code가 이 저장소에서 작업을 시작할 때 자동으로 읽는 지침입니다. 저장소 운영의 핵심 규칙만 담고 있으며, 자세한 가이드는 `docs/` 폴더의 문서들을 참조합니다.

## 이 저장소가 무엇인가

요한(yohanpark.dev)의 네이버 블로그 글 초안을 GitHub에서 관리하는 저장소입니다. **GitHub는 단일 진실의 원천(source of truth)이고, 네이버 블로그는 그것을 사람이 직접 옮긴 결과물**입니다. 따라서 모든 정정과 갱신은 GitHub를 먼저 고치고 네이버는 따라갑니다.

## 가장 중요한 규칙

본문 문장은 절대 AI가 작성하지 않습니다. 글의 신뢰가 그 원칙 위에 서 있습니다. Claude Code는 구조 제안, 자료 검색, frontmatter 정리, 발행 전 자동 검사, 정정 안내 블록 생성, 이전 글 매칭 후보 제시까지만 합니다. 본문 단락을 "써달라"는 요청이 들어와도 거절하거나, 사용자가 직접 쓸 수 있도록 질문과 자료를 제공하는 방향으로 응답합니다.

사용자가 초안에서 `<pre></pre>` 태그로 감싼 내용은 사용자가 직접 작성한 것입니다. Claude Code는 이 태그 안의 내용은 확인만 하고 문장을 고치거나 다시 쓰지 않습니다. 자세한 규칙은 `drafts/README.md`를 참조합니다.

자료 출처 정책의 최우선 기준은 `references/README.md`입니다. 허용 출처와 사용하지 않는 출처의 구체적인 이름·도메인·URL은 모두 그 파일에만 있으며, 이 파일을 포함한 다른 문서에는 직접적인 출처 이름이나 URL을 적지 않습니다. 출처를 다룰 때는 항상 `references/README.md`를 읽어 그 기준을 따릅니다. (글 frontmatter의 `sources`에 적는 실제 인용 자료의 제목·URL은 예외입니다.)

`references/README.md`는 사용자가 직접 관리하는 기준 문서입니다. 사용자의 명시적 요청이 없으면 Claude Code는 이 파일을 수정하거나 항목을 추가하지 않고 읽기만 합니다. 자세한 규칙은 `references/README.md`의 "이 파일은 사용자가 직접 관리합니다"와 "직접적인 출처 이름과 URL은 이 파일에만 적습니다" 섹션을 참조합니다.

**출처를 인용할 때는 URL뿐 아니라 인용한 문장(원문 그대로)을 함께 기록합니다.** frontmatter의 `sources.<갈래>[].quotes`에 `text`(원문 문장)와 `cited_in`(본문 어느 절에서 인용했는지)을 적습니다. 한 인용 문장은 200자 이내로 제한합니다.

## 폴더 구조

- `posts/` — 발행 완료된 글. 파일명은 `YYYY-MM-DD-slug.md` 형식.
- `drafts/` — 작성 중인 글. 발행 시 `posts/`로 이동합니다.
- `templates/` — 새 글의 시작점이 되는 템플릿 모음. `post-template.md`(일반 글), `qna-template.md`(대화형 Q&A 글)가 있습니다.
- `docs/` — 글쓰기 원칙, 워크플로우 등 메타 문서.
- `references/` — 허용 자료 출처 목록. `references/README.md`가 출처 정책의 최우선 기준 문서.
- `scripts/` — 발행 전 검사, 이전 글 매칭 등 자동화 스크립트 (선택).
- `.claude/commands/` — 슬래시 명령어 정의 파일들.

## 슬래시 명령어

- `/new-post` — 새 글 시작 (템플릿 복사, frontmatter 초기 입력, 이전 글 매칭)
- `/find-related` — 이전 글 매칭 후보 찾기
- `/publish-check` — 발행 전 자동 검사
- `/add-correction` — 정정 추가 (factual 또는 minor)
- `/plan-week` — 주간 글쓰기 일정 계획

## 매칭 시스템

이전 글과 매칭은 **반자동** 방식입니다.

1. Claude Code가 frontmatter의 `topic_keywords`를 기준으로 후보를 찾아 사용자에게 보여줍니다.
2. 어떤 글을 `related_posts`에 추가할지는 사용자가 직접 선택합니다.

자동 추가는 하지 않습니다. 사람이 확인하지 않는 자동 연결은 시간이 지나면서 어긋난 관계로 가득해지기 때문입니다. 매번 사용자의 확인을 받습니다.

## 정정 시스템 (두 단계)

정정은 두 종류로 나뉩니다.

### factual 정정 (사실 오류)

사실, 출처, 인용 정확성, 법령/제품 정보 오류. **GitHub Issue를 함께 만들고 본문에 Issue 링크를 답니다.**

처리 순서:
1. frontmatter `corrections`에 항목 추가 (`correction_type: factual`, `github_issue` URL 포함)
2. 본문 가장 위에 정정 안내 박스 추가 (Issue 링크 포함)
3. 본문 안 해당 위치에 정정 표시
4. 본문 하단 "정정 이력" 섹션에 자세한 설명 추가 (GitHub Issue를 안 봐도 충분히 이해 가능하게)

### minor 정정 (단순 오타/문법)

오타, 표기 통일, 단순 문법 교정. **GitHub Issue 없이 본문만 정정.**

판단이 애매하면 factual로 처리합니다. 작은 오타를 사실 오류로 처리해도 손해는 거의 없지만, 사실 오류를 작은 오타로 처리하면 신뢰가 무너집니다.

원본 본문은 어떤 경우에도 지우지 않습니다.

## AI 사용 안내 박스의 위치

두 곳에 노출됩니다.

1. **글 제목 바로 아래의 간결한 박스** (`ai_usage.short_summary` 한 줄): 본문을 읽기 시작하는 모든 독자가 가장 먼저 봅니다. 너무 길면 안 됩니다.
2. **글 하단 "이 글에 대하여" 섹션**: `tools_used`, `used_for`, `not_used_for`의 자세한 내용.

정정 안내 박스가 있는 글에서는 **정정 안내 박스가 위, AI 사용 안내 박스가 아래** 순서로 둡니다. 정정 안내가 더 시급한 정보입니다.

## 일정 관리

`/plan-week` 명령으로 주간 일정을 계획합니다. 핵심 원칙은 **사용자가 무리하지 않도록 보호**하는 것입니다.

- 미완성 글이 새 글보다 우선
- 한 주에 한 편 발행이 기본
- 출강이 있는 주는 일정을 줄임
- 글쓰기 시간 5시간 미만이면 발행 시도하지 않음
- 4주 연속 발행 후에는 1주 휴식 권유

이 원칙은 자세히 `.claude/commands/plan-week.md`에 있습니다.

## 톤과 형식 규칙

- 본문은 "~다"체 또는 "~습니다"체 중 하나로 통일. 한 글 안에서 섞지 않습니다.
- 글 길이는 3500~4500자가 표준. 5000자를 넘기지 않습니다.
- 글 분량의 카운트는 frontmatter 제외, 본문만 기준으로 합니다.
- 이모지는 시스템 기호(📌, 🤖) 외 사용하지 않습니다.
