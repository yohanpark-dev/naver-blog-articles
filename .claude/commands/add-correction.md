# /add-correction — 정정 추가

발행된 글에서 잘못된 내용이 발견되었을 때 정정을 추가하는 명령입니다. 정정 처리의 핵심 원칙은 변하지 않습니다. 원본 본문은 지우지 않으며, 정정 사실은 글 상단에 명확히 노출됩니다.

이 명령은 정정의 무게에 따라 두 가지 경로로 나뉩니다.

- **사실 오류 정정 (factual)**: GitHub Issue를 함께 만들고 본문에 Issue 링크를 답니다. 사실, 출처, 인용 정확성, 법령/제품 정보 오류가 여기 해당합니다.
- **단순 정정 (minor)**: GitHub Issue 없이 본문만 정정합니다. 오타, 단순 문법 교정, 표기 통일이 여기 해당합니다.

## 1. 정정의 무게 확인

먼저 사용자에게 정정의 무게를 묻습니다.

> 이 정정은 어느 쪽에 해당하나요?
> 1. 사실 오류 / 출처 오류 / 인용 부정확 → GitHub Issue를 함께 만듭니다
> 2. 단순 오타 / 표기 교정 → GitHub Issue 없이 본문만 고칩니다

판단이 애매한 경우(예: 일반 표현이 잘못된 정보를 줄 위험이 있는 경우)는 1번을 선택하는 것을 권합니다. 신뢰의 비대칭성 때문입니다. 작은 오타도 사실 오류로 처리해서 손해는 거의 없지만, 사실 오류를 작은 오타로 처리하면 신뢰가 무너집니다.

## 2. 사용자에게 묻기

다음을 한 번에 모아 묻습니다.

- 어느 글에 대한 정정인가요? (파일 경로 또는 글 제목)
- 무엇이 틀렸나요? (본문의 어느 문장이 잘못되었는지 인용)
- 무엇이 맞나요? (정확한 내용. 1~2 문장으로 압축)
- 잘못된 이유는? (예: 법령 개정을 인지하지 못함, 공식 문서를 잘못 읽음, Copilot 기능 업데이트로 동작이 바뀜)
- 근거 자료 URL은? (factual인 경우 필수. 허용 도메인 검증)

근거 자료 URL이 허용 도메인 외(블로그, 커뮤니티, 언론사 등)라면 진행을 보류하고 다른 출처를 요청합니다.

## 3. Frontmatter에 정정 이력 추가

대상 글의 frontmatter `corrections` 배열에 다음 형식으로 추가합니다.

### factual 정정 (GitHub Issue 포함)

```yaml
corrections:
  - date: "2026-08-15"
    correction_type: "factual"
    what: "본문 4절 ②에서 OO법 X조라고 적었으나 Y조의 오기"
    correct: "OO법 Y조 — 외부 클라우드 AI 응답에 대한 신용정보 제한 규정"
    reason: "법령 개정 사실을 잘못 인지"
    source: "https://www.law.go.kr/..."
    github_issue: "https://github.com/USERNAME/yohanpark-blog/issues/12"
```

### minor 정정 (Issue 없음)

```yaml
corrections:
  - date: "2026-08-15"
    correction_type: "minor"
    what: "본문 2절에서 '카피로트'를 'Copilot'으로 표기 통일"
    correct: "Copilot"
    reason: "표기 통일"
```

기존 정정 이력이 있으면 그 아래에 새 항목을 추가합니다. 시간 순서로 정렬되어야 합니다.

## 4. GitHub Issue 생성 (factual 정정만)

correction_type이 factual인 경우, 다음 정보로 GitHub Issue를 만들 것을 사용자에게 제안합니다. Linear MCP가 GitHub과 연결되어 있다면 자동 생성을 시도하고, 아니면 다음 내용을 사용자에게 보여주고 직접 만들도록 안내합니다.

```
제목: [정정] [글 제목] - [무엇이 잘못되었는지 한 줄 요약]

본문:
## 어느 글인가
- 글 제목: [title]
- 글 URL: [naver_url]
- 파일: [파일 경로]

## 무엇이 잘못되었나
[본문의 잘못된 문장과 무엇이 틀렸는지]

## 무엇이 맞나
[정확한 내용]

## 왜 잘못되었나
[작성 시점에 왜 이 오류가 발생했는지]

## 근거 자료
[URL과 핵심 인용 문장]

## 처리 결과
- [x] 본문 정정 표시 추가
- [x] 정정 이력 섹션에 자세한 설명 추가
- [x] 네이버 블로그에도 동일하게 반영 (별도 처리)
- 정정일: [date]
```

Issue가 만들어지면 그 URL을 frontmatter의 `github_issue` 필드에 적습니다.

## 5. 본문 상단에 정정 안내 박스 노출

대상 글의 본문 가장 위(제목 바로 아래, AI 사용 안내 박스 위)에 정정 안내 박스를 추가합니다.

```markdown
> **📌 정정 안내**
>
> 이 글에는 정정된 내용이 있습니다.
> - **2026-08-15 정정**: 본문 4절 ②의 법령 조항 오류를 바로잡았습니다. ([상세 보기](#정정-이력) · [관련 GitHub Issue](https://github.com/USERNAME/yohanpark-blog/issues/12))
```

minor 정정의 경우 `· [관련 GitHub Issue](...)` 부분은 생략합니다.

이미 정정 안내 박스가 있으면 새 정정만 목록에 추가합니다. 시간순으로 가장 최근 정정이 위에 오도록 정렬합니다.

## 6. 본문 안 해당 위치에 정정 표시

본문의 잘못된 문장 바로 뒤(또는 단락 끝)에 다음 표시를 답니다. 원래 문장은 지우지 않습니다.

```markdown
[원래 문장은 그대로]

> ※ **2026-08-15 정정** — 위 문장의 [무엇] 부분이 잘못되었습니다. 정확한 내용은 [정정 이력](#정정-이력) 참조.
```

## 7. 본문 하단 "정정 이력" 섹션 갱신

본문 가장 아래에 "정정 이력" 섹션이 있는지 확인하고, 없으면 추가, 있으면 새 정정을 풀어 적습니다. **GitHub Issue를 안 보고 이 섹션만 읽어도 충분히 이해되도록 자세히 씁니다.** GitHub에 익숙하지 않은 독자가 이 섹션의 1차 독자입니다.

```markdown
## 정정 이력

### 2026-08-15 정정 (사실 오류)

- **무엇이 잘못되었나**: 본문 4절 ②에서 인용한 OO법 X조는 실제로는 Y조입니다.
- **무엇이 맞나**: OO법 Y조의 내용은 [정확한 조문 내용을 풀어서 적음].
- **왜 잘못되었나**: 법령 개정 사실을 인지하지 못한 채 작성했습니다. 한 독자분께서 댓글로 지적해주셨습니다.
- **무엇을 참고했나**: [법령정보센터 — OO법](URL)
- **GitHub 이슈**: [관련 논의 보기](https://github.com/USERNAME/yohanpark-blog/issues/12) (개발자가 아니어도 누구나 열람 가능합니다)
```

minor 정정의 경우 GitHub 이슈 줄을 생략합니다.

## 8. 마무리

작업이 끝나면 다음을 사용자에게 보고합니다.

- 정정된 파일 경로
- frontmatter `corrections`에 추가된 항목
- 본문 상단 정정 안내 박스 갱신 위치
- 본문 안 정정 표시 위치
- 본문 하단 정정 이력 섹션 갱신 내용
- (factual인 경우) GitHub Issue URL

그 후 다음과 같이 안내합니다.

> 다음으로 네이버 블로그에도 동일하게 반영해주세요. 순서는 다음을 권합니다.
> 1. 본문 상단 정정 안내 박스를 가장 먼저 추가
> 2. 본문 안 해당 문장 옆에 정정 표시 추가
> 3. 본문 하단 정정 이력 섹션 갱신
>
> GitHub는 소스 파일이고 네이버 블로그는 그 결과물입니다. 두 곳의 내용이 어긋나지 않도록 가능한 빨리 반영하시기를 권합니다.
