# /publish-check — 발행 전 검사

발행 직전 글을 자동으로 검사하는 명령입니다. 문제가 발견되면 보고만 하고, 사용자 확인 없이 본문이나 frontmatter를 고치지 않습니다.

## 1. 대상 확인

인자로 파일 경로가 주어졌다면 그 파일을, 아니라면 `drafts/` 안에서 가장 최근에 수정된 파일을 대상으로 합니다.

## 2. 검사 항목

다음 항목을 순서대로 검사하고, 각 항목에 대해 통과/실패와 이유를 보고합니다.

### A. Frontmatter 필수 항목

다음 항목이 비어 있지 않은지 확인합니다.

- title, slug, date_created, category, tags, topic_keywords, volatility
- ai_usage.short_summary, ai_usage.tools_used, ai_usage.used_for, ai_usage.not_used_for
- test_environment.plan, test_environment.period, test_environment.os, test_environment.apps_tested

하나라도 비어 있으면 어느 항목이 비었는지 명시합니다.

### B. 출처 형식 검증 (새 요구사항)

`sources` 배열의 각 항목(microsoft_official, legal, security_vendor)에 대해 다음을 검사합니다.

#### B-1. URL 도메인 검증

각 출처의 URL이 허용 도메인인지 확인합니다.

- `microsoft_official`: `learn.microsoft.com/ko-kr/`로 시작해야 합니다. 영문 페이지(`/en-us/`)나 Microsoft 블로그는 거부합니다.
- `legal`: `law.go.kr`, `*.go.kr`, 또는 명시적으로 공공기관으로 등록된 도메인.
- `security_vendor`: frontmatter에 등록되거나 일반적으로 인정되는 보안 솔루션 업체 공식 도메인.

#### B-2. 인용 문장(quotes) 존재 검증

각 출처 항목에 `quotes` 배열이 있고, 그 안에 최소 한 개의 `text`와 `cited_in`이 채워져 있는지 확인합니다. 출처를 frontmatter에 등록했는데 인용 문장이 없으면 다음과 같이 보고합니다.

> `sources.microsoft_official[0]` ("[문서 제목]")에 인용 문장(quotes)이 없습니다. 본문에서 이 자료를 어떻게 인용했는지 원문 문장을 함께 기록해야 합니다.

#### B-3. 인용 문장 길이 검증

각 `quotes.text`가 너무 길지 않은지 확인합니다. 한 인용 항목이 200자를 초과하면 경고합니다.

> `sources.microsoft_official[0].quotes[0]`의 인용 문장이 [N]자입니다. 정당한 인용 범위를 위해 200자 이내로 압축하시기를 권합니다.

#### B-4. 본문과 출처의 cited_in 일치 검증

각 `quotes.cited_in`("본문 3절" 등)에 해당하는 섹션이 본문에 실제로 존재하는지 확인합니다. 없으면 다음과 같이 보고합니다.

> `sources.microsoft_official[0].quotes[0].cited_in`이 "본문 3절"을 가리키지만, 본문에 해당 절이 없습니다.

#### B-5. 본문 안 URL 검증

본문에서 발견되는 모든 URL을 추출해 허용 도메인 외 URL이 있는지 확인합니다. 특히 다음 도메인은 명확히 금지이므로 강하게 경고합니다.

- 네이버 블로그, 티스토리, 브런치, 미디엄, 개인 블로그
- 커뮤니티 사이트(클리앙, 디씨, 레딧 등)
- 언론사 도메인
- learn.microsoft.com이지만 `/ko-kr/`이 아닌 경우

### C. AI 클리셰 검사

본문에서 다음 표현을 찾아 줄 번호와 함께 보고합니다.

- "~에 대해 알아보겠습니다"
- "결론적으로"
- "오늘은 ~에 대해"
- "한 번 살펴보겠습니다"
- "여러분"
- "다양한" (한 글에 3회 이상 등장하면 경고)
- "이번 시간에는"
- "함께 알아보아요"

표현을 자동으로 수정하지 않습니다. 사용자가 직접 판단해서 고치도록 위치와 표현만 알립니다.

### D. 본문 길이

frontmatter를 제외한 본문의 글자 수(공백 포함)를 셉니다. 권장 범위는 3500~4500자입니다. 범위를 벗어나면 경고하되, 사용자가 의도적으로 그 길이를 선택했을 수 있으므로 강제하지 않습니다.

### E. 이전 글 매칭 확인

frontmatter의 `topic_keywords`로 `/find-related`를 내부적으로 실행해서, `related_posts`에 등록되지 않은 강한 후보가 있는지 확인합니다. 있으면 "다음 글이 비슷한 주제이지만 `related_posts`에 등록되어 있지 않습니다 — 검토하셨는지 확인해 주세요"라고 안내합니다.

### F. 톤 일관성

본문에서 "~다."로 끝나는 문장 수와 "~습니다."로 끝나는 문장 수를 셉니다. 한쪽이 80% 이상이면 통과, 50:50에 가까우면 톤 검토를 권합니다.

### G. AI 사용 안내 박스 존재 확인

본문 시작 부분(제목 바로 아래 또는 정정 안내 박스 아래)에 AI 사용 안내 박스가 있는지 확인합니다. 박스가 없거나 frontmatter의 `ai_usage.short_summary`와 박스 내용이 불일치하면 경고합니다.

### H. 정정 안내 박스 일관성

frontmatter의 `corrections` 배열에 항목이 있는데 본문 가장 위에 정정 안내 박스가 노출되지 않은 경우, 또는 그 반대의 경우를 감지해 경고합니다. `correction_type: factual`인 정정이 있는데 `github_issue` 필드가 비어 있으면 함께 경고합니다.

### I. "이 글에 대하여" 섹션 존재

본문 마지막에 "이 글에 대하여" 섹션이 있고, 그 안에 사용한 AI 도구 / 검증 환경 / 출처 세 항목이 모두 있는지 확인합니다.

## 3. 보고 형식

각 항목에 대해 통과/실패를 표시한 요약을 먼저 보여주고, 실패한 항목의 상세 내용을 그 아래에 풀어 적습니다. 예시:

```
발행 전 검사 결과

✓ A. Frontmatter 필수 항목
✓ B-1. URL 도메인
✗ B-2. 인용 문장 존재 — 1건
✓ B-3. 인용 길이
✓ B-4. cited_in 일치
✓ B-5. 본문 안 URL
✓ C. AI 클리셰
⚠ D. 본문 길이 — 권장 범위 벗어남 (현재 4800자)
✓ E. 이전 글 매칭
✓ F. 톤 일관성
✓ G. AI 안내 박스
✓ H. 정정 안내 일관성
✓ I. "이 글에 대하여" 섹션

[상세]
B-2: sources.security_vendor[0] ("OO업체 백서")에 인용 문장이 없습니다.
D: 본문 4800자로 권장 범위(3500~4500) 초과. 의도한 길이라면 발행 가능.

[결론] 발행 전 1건 수정 필요 (B-2)
```

사용자에게 어떤 항목을 어떻게 고칠지 제안만 하고, 직접 고치지 않습니다.

마지막에 "발행 가능 / 발행 전 수정 필요"를 한 줄로 결론냅니다.
