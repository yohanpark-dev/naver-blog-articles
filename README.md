# yohanpark-blog

요한(yohanpark.dev)의 네이버 블로그 글 초안을 관리하는 저장소입니다.

## 이 저장소가 무엇인가

마크다운으로 블로그 글을 쓰고 네이버 블로그에 발행하는 작업의 작은 운영 시스템입니다. **GitHub는 단일 진실의 원천(source of truth)이고, 네이버 블로그는 사람이 직접 옮긴 컴파일 결과물**입니다. 모든 정정과 갱신은 GitHub를 먼저 고치고 네이버는 따라갑니다.

본문 문장은 사람이 직접 씁니다. AI는 구조 설계, 자료 조사, 정정 자동화, 이전 글 매칭, 발행 전 검사 같은 보조 작업만 합니다. 이 원칙은 `.claude/CLAUDE.md`에 못박혀 있으며 어떤 명령으로도 우회되지 않습니다.

## 폴더 구조

`posts/` 폴더에는 발행이 완료된 글이 있습니다. 파일명은 `YYYY-MM-DD-slug.md` 형식입니다. `drafts/` 폴더는 작성 중인 글의 공간이고, 발행 시점에 `posts/`로 옮겨집니다.

`templates/` 폴더에는 새 글의 시작점이 되는 템플릿이 있습니다. `post-template.md`는 일반 글(리뷰, 사용기, 정리 글)의 기본 템플릿이고, `qna-template.md`는 저자와 제3의 가상 인물이 질문·답변을 주고받는 대화형 Q&A 글 템플릿입니다. Claude Code의 `/new-post` 명령으로 선택한 템플릿이 자동 복사됩니다.

`docs/` 폴더에는 글쓰기 원칙, frontmatter 스키마, 워크플로우 같은 메타 문서가 있습니다. 자주 보지는 않지만 시간이 지나 규칙을 잊었을 때 다시 찾아보는 자리입니다.

`scripts/` 폴더는 자동화 스크립트 자리로 비어 있어도 됩니다. Claude Code의 슬래시 명령으로 같은 일을 더 유연하게 할 수 있습니다.

`CODEX.md`는 Codex가 이 저장소에서 작업할 때 빠르게 참고할 운영 요약입니다. Claude Code 지침과 같은 원칙을 따르되, Codex 작업 흐름에 맞춰 압축해 둔 문서입니다.

`.claude/` 폴더는 Claude Code가 자동으로 읽는 영역이며, `CLAUDE.md`(작업 지침)와 `commands/` (슬래시 명령어들)가 들어 있습니다.

## 슬래시 명령어

Claude Code에서 다음 명령을 호출해 작업을 자동화합니다.

`/plan-week`은 한 주 일정을 계획하는 명령입니다. 현재 진행 중인 글과 사용자의 다음 주 일정을 함께 보고 무리하지 않는 일정을 제안합니다. 매주 일요일 저녁이나 월요일 아침에 호출합니다.

`/new-post`는 새 글을 시작하는 명령입니다. 시작 템플릿, 제목, 분류, 키워드 등을 한 번에 묻고, 이전 글 중 비슷한 주제가 있으면 자동으로 찾아 보여줍니다. 어떤 글을 관련 글로 연결할지는 사용자가 직접 결정합니다.

`/find-related`는 이전 글 매칭 후보를 찾는 명령입니다. 작성 중인 글에 어떤 이전 글이 연결되어야 할지 검토할 때 사용합니다.

`/publish-check`는 발행 직전 자동 검사입니다. frontmatter 완성도, 출처 도메인, 인용 문장 형식, AI 클리셰, 본문 길이, 톤 일관성 등을 모두 검사합니다.

`/add-correction`은 발행된 글에 정정을 추가하는 명령입니다. 정정의 무게에 따라 두 경로(factual / minor)로 나뉘어 처리됩니다.

## 핵심 시스템 개요

### 매칭 시스템 (반자동)

이전 글과의 매칭은 두 단계로 작동합니다. Claude Code가 `topic_keywords` 기준으로 후보를 찾아 보여주고, 어떤 글을 `related_posts`에 추가할지는 사용자가 명시적으로 선택합니다. AI 자동 매칭에 사람 확인 없이 의존하지 않는 것이 시스템의 일관성을 만듭니다.

### 정정 시스템 (두 단계)

정정은 무게에 따라 두 종류로 처리됩니다.

**사실 오류 정정 (factual)**: 사실, 출처, 인용 정확성 오류. GitHub Issue를 함께 만들고 본문 정정 안내에 Issue 링크를 답니다. 본문 하단의 "정정 이력" 섹션은 Issue를 안 봐도 충분히 이해 가능하게 자세히 적습니다.

**단순 정정 (minor)**: 오타, 표기 통일. GitHub Issue 없이 본문만 정정.

원본 본문은 어떤 경우에도 지우지 않습니다. 정정 사실은 글 상단에 명확히 노출됩니다.

### AI 사용 투명성

각 글의 frontmatter에는 `ai_usage` 항목이 있고, 그 내용이 본문 두 곳에 노출됩니다.

- 글 제목 바로 아래의 간결한 박스 (한 줄 요약)
- 글 하단 "이 글에 대하여" 섹션 (자세한 내용)

숨기지 않고 드러내는 것이 가장 정직한 포지셔닝이며, 장기적으로 가장 큰 신뢰 자산이 됩니다.

### 자료 출처 정책

이 저장소에서 허용되는 자료 출처는 `references/README.md`가 최우선 기준입니다. 다른 문서의 출처 설명은 요약이므로, 충돌이 생기면 `references/README.md`를 따릅니다.

현재 기준상 블로그, 커뮤니티, 언론 기사는 사용하지 않습니다. 이 규칙은 `/publish-check`에서 자동 검사됩니다.

각 출처를 인용할 때는 URL뿐 아니라 **원문 문장 그대로**를 frontmatter의 `quotes`에 함께 기록합니다. 본문에서는 자기 문장과 원문 인용이 짝을 이루는 참조 블록 형식을 사용합니다.

### 두 겹 포지션

글의 신뢰는 두 시점의 교차에서 만들어집니다. 하나는 yohanpark.dev에서 직접 도구를 써본 개인 사용자의 시점, 다른 하나는 기업 교육 강사로서 여러 회사의 도입 사례를 관찰한 시점입니다. 두 시점이 본문에서 자연스럽게 교차해야 합니다.

## 일반적인 작업 흐름

매주 시작 시점에 `/plan-week`로 일정을 잡습니다. 새 글을 시작할 때는 `/new-post`로 파일을 만들고, 자료 조사를 하면서 frontmatter `sources`에 URL과 원문 인용을 함께 적어둡니다. 본문은 직접 쓰고, 그동안 Claude Code에게 자료 검색이나 구조 점검을 위임할 수 있습니다. 본문이 완성되면 `/publish-check`로 검사를 돌리고, 통과되면 네이버 블로그로 옮깁니다. 발행 후 사실 오류가 발견되면 `/add-correction`을 호출합니다.
