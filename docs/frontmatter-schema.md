# Frontmatter 스키마

각 글의 맨 위에 들어가는 frontmatter는 단순한 주석이 아니라 시스템이 작동하기 위한 구조화된 데이터입니다. 이 문서는 각 필드가 무엇을 의미하고, 어떤 값을 받아야 하며, 잘못 채웠을 때 어떤 문제가 생기는지를 항목별로 설명합니다.

## 식별 정보

`title`은 글의 제목입니다. 네이버 블로그에 그대로 들어가는 텍스트이므로, 이 자리에는 한국어 완성형 문장을 적습니다. 이모지나 특수문자는 쓰지 않습니다. 제목은 검색 유입의 핵심이므로 핵심 키워드가 자연스럽게 들어가야 합니다.

`slug`는 파일명과 URL에 쓰이는 영문 식별자입니다. 영문 소문자와 하이픈만 사용하며, 공백이나 한글은 쓰지 않습니다. 예: `ms365-copilot-first-month`. 이 식별자가 글의 고유 ID 역할을 하기 때문에, 한 번 정한 뒤에는 바꾸지 않는 것이 좋습니다. 매칭 시스템이 이 값으로 글을 식별하기 때문에, 바꾸면 다른 글의 `related_posts` 참조가 깨집니다.

`date_created`는 글을 시작한 날짜이고, `date_published`는 네이버 블로그에 발행한 날짜입니다. 발행 전에는 `date_published`를 비워둡니다. `status` 값은 `draft`로 시작해서 검토 단계로 가면 `review`, 발행 후에는 `published`, 의도적으로 더 이상 노출하지 않을 때는 `archived`로 바뀝니다.

## 분류

`category`는 글의 큰 분류입니다. `tech-review`, `work-tool`, `education` 같은 영문 식별자를 씁니다. 카테고리는 자주 늘어나지 않는 것이 좋습니다.

`tags`는 네이버 블로그 태그용입니다. 한국어로 적으며, 네이버 검색에서 잡힐 만한 단어들을 8~10개 정도 넣습니다. 예: `["MS365", "마이크로소프트365", "Copilot", "코파일럿 후기", "MS365 보안", "기업 AI 도입"]`.

`topic_keywords`는 `tags`와 다른 목적을 가진 별도의 항목입니다. 매칭 시스템이 사용하는 키워드이며, 같은 글의 `tags`보다 더 좁은 단위입니다. 한 글에 2~4개 정도가 적당하며, 영문 케밥 케이스(소문자-하이픈)로 적습니다. 좋은 예시는 `copilot-korean-quality`, `ms365-licensing-business`, `enterprise-ai-governance` 같은 입자도입니다.

## 글의 수명

`volatility`는 글의 정보가 얼마나 빨리 낡는지를 표시합니다. 세 가지 값 중 하나를 가집니다.

`high`는 기능, 가격, UI 등 빠르게 바뀌는 정보를 다루는 글입니다. 3~6개월 후 점검 대상이 됩니다.

`medium`은 사용 경험과 평가가 중심인 글입니다. 단편 사용기 대부분이 여기 해당합니다. 1년 정도면 점검을 한 번 해보는 것이 좋습니다.

`low`는 원칙, 거버넌스, 법적 근거 같은 잘 안 낡는 정보를 다루는 글입니다.

이 항목은 발행 직후에는 사용되지 않습니다. 6개월~1년이 지나면서 작동하기 시작하는 장치입니다. Claude Code에게 "최근 6개월 동안 점검하지 않은 high 볼라틸리티 글이 뭐가 있지?"라고 물었을 때 답을 받기 위한 메타데이터입니다.

## 다른 글과의 관계

`related_posts`는 이 글과 관련된 이전 글들의 slug 목록입니다. Claude Code의 `/find-related` 명령이 후보를 찾아주지만, 어떤 글을 실제로 연결할지는 사람이 결정합니다. 자동 추가되지 않습니다. 사람이 확인하지 않는 자동 연결은 시간이 지나면서 어긋난 관계로 가득해지기 때문입니다.

`supersedes`와 `superseded_by`는 같은 주제를 다시 다루는 경우의 양방향 연결입니다.

## 정정 이력

`corrections`는 정정이 발생할 때마다 추가되는 배열입니다. 두 종류의 정정을 구분해서 기록합니다.

### factual 정정 (사실 오류)

사실, 출처, 인용 정확성, 법령/제품 정보 오류가 여기 해당합니다. GitHub Issue를 함께 만들고 본문에 Issue 링크를 답니다.

```yaml
corrections:
  - date: "2026-08-15"
    correction_type: "factual"
    what: "본문 4절 ②에서 인용한 OO법 X조는 실제로는 Y조"
    correct: "OO법 Y조 — 외부 클라우드 AI 응답에 대한 신용정보 제한 규정"
    reason: "법령 개정 사실을 인지하지 못함"
    source: "https://www.law.go.kr/..."
    github_issue: "https://github.com/USERNAME/yohanpark-blog/issues/12"
```

### minor 정정 (단순 오타/문법)

오타, 표기 통일, 단순 문법 교정이 여기 해당합니다. GitHub Issue 없이 본문만 정정합니다.

```yaml
corrections:
  - date: "2026-08-15"
    correction_type: "minor"
    what: "본문 2절에서 '카피로트'를 'Copilot'으로 표기 통일"
    correct: "Copilot"
    reason: "표기 통일"
```

판단이 애매한 경우에는 factual로 처리합니다. 작은 오타를 사실 오류로 처리해도 손해는 거의 없지만, 사실 오류를 작은 오타로 처리하면 신뢰가 무너집니다.

이 항목들은 모두 `/add-correction` 명령으로 자동 추가됩니다. 수동으로 본문만 고치면 frontmatter와 본문이 어긋나게 되므로, 정정은 항상 명령을 통해 처리합니다.

## AI 사용 기록

`ai_usage`는 글의 신뢰를 만드는 핵심 항목입니다. 네 하위 필드가 있습니다.

`short_summary`는 글 제목 바로 아래에 노출되는 박스의 내용입니다. 한 줄(60자 이내) 정도로 압축합니다. 예: "구조 설계와 자료 정리에 Claude 사용. 본문은 직접 작성." 이 문장은 본문을 읽기 시작하는 모든 독자가 가장 먼저 보는 메타정보이므로, 너무 길면 본문이 시작되기도 전에 메타정보에 지치게 됩니다.

`tools_used`에는 글 작성에 사용한 AI 도구를 적습니다. 도구 이름과 사용 목적을 함께 적습니다. 예: `["Claude (Anthropic) — 글 구조 설계", "Claude Code — 자료 정리와 발행 전 검사"]`. 이 내용은 글 하단 "이 글에 대하여" 섹션에 자세히 노출됩니다.

`used_for`에는 AI를 사용한 구체적인 작업을 나열합니다. 예: `["글 구조 설계", "공식 문서 검색과 정리", "발행 전 자동 검사", "맞춤법 검토"]`.

`not_used_for`는 AI를 사용하지 않은 영역을 명시하는 항목입니다. 이 항목이 가장 중요합니다. 독자에게 "본문은 사람이 직접 썼다"는 메시지를 전달하는 부분이기 때문입니다. 예: `["본문 문장 작성", "주관적 평가와 결론", "기업 현장 사례"]`.

이 네 항목이 모두 채워져야 합니다. 비어 있으면 발행 전 검사에서 실패합니다.

## 출처

`sources`는 세 갈래로 분리됩니다. 그리고 각 출처 항목 안에 `quotes` 배열이 있어 본문에서 인용한 문장을 함께 기록합니다. 이 형식이 신뢰의 핵심입니다.

```yaml
sources:
  microsoft_official:
    - title: "Microsoft 365 Copilot의 데이터, 개인 정보 및 보안"
      url: "https://learn.microsoft.com/ko-kr/copilot/microsoft-365/microsoft-365-copilot-privacy"
      quotes:
        - text: "Microsoft 365 Copilot은 사용자의 프롬프트 또는 응답을 기반 모델을 학습하는 데 사용하지 않습니다."
          cited_in: "본문 3절"
        - text: "사용자가 액세스 권한이 있는 데이터에만 응답을 생성합니다."
          cited_in: "본문 4절 ③"
```

### 각 갈래의 허용 범위

`microsoft_official`에는 `learn.microsoft.com/ko-kr/` 도메인의 한국어 페이지만 들어갑니다. 영문 페이지나 Microsoft 블로그는 허용되지 않습니다. 발행 전 검사에서 도메인이 자동으로 검증됩니다.

`legal`에는 법령정보센터(`law.go.kr`)와 공공기관 가이드라인이 들어갑니다. 다른 도메인은 명시적으로 등록하지 않으면 거부됩니다.

`security_vendor`에는 보안 솔루션 업체의 공식 페이지나 백서가 들어갑니다.

### quotes 항목의 규칙

`quotes` 배열의 각 항목은 `text`와 `cited_in` 두 필드를 가집니다.

`text`는 일차 자료에서 그대로 가져온 문장입니다. 임의로 줄이거나 의역하지 않습니다. 의역해야 한다면 그것은 본문에 풀어 쓰는 자기 문장이지 인용이 아닙니다.

한 인용 문장의 길이는 200자 이내로 제한합니다. 정당한 인용 범위를 지키기 위함입니다. 한 문서에서 여러 곳을 인용하려면 quotes 배열에 항목을 여러 개 두되, 한 문서당 합산 인용량이 한 단락(400자) 이내가 되도록 합니다.

`cited_in`은 이 인용이 본문 어디에서 사용되었는지를 표시합니다. 예: "본문 3절", "본문 4절 ③". 발행 전 검사에서 이 값이 가리키는 섹션이 본문에 실제로 존재하는지 자동 검증됩니다.

## 검증 환경

`test_environment`는 사용기 글에서 특히 중요합니다. 독자가 "이 사람의 검증 결과가 내 환경에도 적용될까?"를 판단하는 근거가 됩니다.

`plan`에는 사용한 MS365 플랜과 Copilot 라이선스를 정확히 적습니다. 예: "Microsoft 365 Business Standard + Microsoft 365 Copilot". `period`는 검증 기간을 YYYY-MM 형식으로 적습니다. `domain`은 yohanpark.dev로 고정되어 있습니다. `os`에는 macOS나 Windows의 정확한 버전을 적습니다. `apps_tested`에는 한 달 동안 실제로 검증한 앱들을 나열합니다.

## 일정 관리

`linear_issue_id`는 Linear에서 만든 이슈의 ID입니다. 예: `BLOG-23`. `/plan-week` 명령으로 일정을 잡을 때 자동으로 이슈가 만들어지고 이 필드가 채워집니다.

## 발행 정보

`naver_url`은 발행이 완료된 후 네이버 블로그의 글 URL을 적는 자리입니다. 발행 직후 채웁니다.

## 필드를 새로 추가하고 싶을 때

새 필드가 필요하다고 느낄 때는 먼저 다음을 검토합니다. 첫째, 이 정보가 다른 글과의 매칭, 검색, 자동화 중 어디에 쓰이는지. 둘째, 이 정보가 글마다 달라지는지(그렇다면 frontmatter), 아니면 모든 글에 공통인지(그렇다면 시스템 설정). 셋째, 사람이 직접 채우는지, 자동으로 채워지는지.

새 필드를 추가하면 이 문서에도 항목 설명을 추가하고, `.claude/CLAUDE.md`의 발행 전 검사 항목에도 필요하면 추가합니다.
