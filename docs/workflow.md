# 워크플로우

이 문서는 글 한 편이 만들어지고 발행되고 시간이 지나 정정되거나 후속 글로 이어지는 전체 흐름을 시간 순서대로 설명합니다.

## 단계 0 — 주간 일정 계획

매주 일요일 저녁 또는 월요일 아침, Claude Code에서 `/plan-week`를 실행합니다. 이 명령은 현재 진행 중인 글, 최근 발행 이력, 사용자의 다음 주 일정을 모두 확인한 다음 무리하지 않는 일정을 제안합니다.

이 단계는 단순한 일정 계획이 아니라 **사용자가 무리하지 않도록 보호하는 장치**입니다. 미완성 글이 누적되면 새 글을 시작하지 말라고 권하고, 출강이 있는 주는 발행 일정을 줄이고, 4주 연속 발행 후에는 한 주 휴식을 권합니다. 일정대로 안 됐다고 자책할 필요는 없습니다. 다음 주에 다시 잡으면 됩니다.

## 단계 1 — 새 글 주제와 파일 생성

글의 시작은 주제를 떠올리는 일에서 출발합니다. 이 단계는 시스템화하기보다 자유롭게 두는 것이 좋습니다. 다만 떠오른 주제가 글이 될 가치가 있는지 판단할 때는 두 가지 질문을 던집니다. 첫째, 이 주제에 대해 yohanpark.dev에서 직접 써본 경험이 있는가, 또는 기업 교육 현장에서 본 사례가 있는가. 둘째, learn.microsoft.com 한국어 페이지에서 일차 자료를 찾을 수 있는가.

주제가 정해지면 Claude Code에서 `/new-post`를 실행합니다. Claude Code가 제목, slug, 분류, 키워드, 검증 환경을 한 번에 묻고, 답을 받은 즉시 다음 작업을 합니다. 첫째, 이전 글 중 비슷한 주제를 가진 글을 찾아 후보를 보여줍니다(매칭). 어떤 글을 관련 글로 연결할지는 사용자가 직접 결정합니다. 둘째, `templates/post-template.md`를 `drafts/오늘날짜-slug.md`로 복사합니다. 셋째, frontmatter의 채울 수 있는 항목을 자동으로 채웁니다. 본문은 손대지 않습니다.

`/plan-week`에서 만들어진 Linear 이슈가 있다면 그 ID가 frontmatter의 `linear_issue_id`에 자동으로 들어갑니다.

## 단계 2 — 자료 조사

본문을 쓰기 전에 자료를 먼저 정리하는 것이 효율적입니다. 자료가 부족한 상태로 본문을 쓰기 시작하면 중간에 막혀서 자료를 찾으러 나가게 되고, 흐름이 끊깁니다.

자료 조사는 사용자가 직접 하거나 Claude Code에게 위임할 수 있습니다. Claude Code에게 시킬 때는 "Microsoft 365 Copilot의 데이터 처리 정책을 learn.microsoft.com 한국어 페이지에서 찾아서 핵심 내용을 정리해줘" 같은 식으로 요청합니다.

자료를 찾을 때마다 frontmatter의 `sources` 항목에 즉시 추가합니다. 이때 URL뿐 아니라 **인용할 원문 문장**을 함께 적습니다. 본문에서 그 문장을 어느 절에서 인용할지는 본문을 쓰면서 결정하지만, 인용할 문장 자체는 자료 조사 단계에서 미리 골라둡니다. 본문을 다 쓰고 출처를 정리하려고 하면 어느 문장을 인용했는지 헷갈리므로, 자료를 찾는 즉시 quotes에 적어두는 것이 좋습니다.

```yaml
sources:
  microsoft_official:
    - title: "Microsoft 365 Copilot의 데이터, 개인 정보 및 보안"
      url: "https://learn.microsoft.com/ko-kr/copilot/..."
      quotes:
        - text: "Microsoft 365 Copilot은 사용자의 프롬프트 또는 응답을 기반 모델을 학습하는 데 사용하지 않습니다."
          cited_in: "본문 3절"  # 본문 작성 후에 채워도 됨
```

## 단계 3 — 본문 작성

본문은 사용자가 직접 씁니다. AI에게 본문 단락을 써달라고 요청하지 않습니다. 이 원칙은 시스템 전체의 신뢰 근거이므로 어떤 경우에도 어기지 않습니다.

본문을 쓰는 중에 도움이 필요한 경우는 여러 가지가 있을 수 있습니다. 자료를 다시 찾아야 할 때는 Claude Code에게 추가 검색을 요청합니다. 한 섹션을 다 쓴 다음 구조가 어색한 것 같으면 Claude Code에게 "이 섹션의 구조에 대해 의견을 줘"라고 물어볼 수 있습니다. 단, 의견은 받지만 수정은 사용자가 직접 합니다.

본문에서 일차 자료를 인용할 때는 다음 형식을 따릅니다.

```
> **참조**: [본문의 자기 문장 한 줄 — 인용한 내용을 자기 말로 풀어 쓴 것]
> 원문: "[일차 자료에서 그대로 가져온 한두 문장]"
> 출처: [문서 제목](URL)
```

이 형식이 본문 문장 + 원문 인용 + 출처 링크를 한 자리에 모아 신뢰감을 만듭니다. 인용 문장은 200자 이내로 짧게 씁니다.

본문을 쓰면서 두 겹 포지션(개인 사용자 + 기업 교육 강사)이 자연스럽게 교차하고 있는지 의식합니다. 한쪽으로 너무 기울고 있으면 의식적으로 다른 시점의 사례를 끼워 넣습니다.

## 단계 4 — 발행 전 검사

본문이 완성되면 Claude Code에서 `/publish-check`를 실행합니다. 검사 항목은 다음과 같습니다.

- Frontmatter 필수 항목 모두 채워짐
- 출처 URL이 허용 도메인에 속함 (`learn.microsoft.com/ko-kr/` 등)
- 각 출처에 `quotes` 인용 문장이 함께 기록됨
- 인용 문장이 200자 이내
- `quotes.cited_in`이 가리키는 본문 섹션이 실제로 존재함
- 본문 안 URL이 허용 도메인 외 출처를 포함하지 않음
- AI 클리셰 표현이 본문에 남아 있지 않음
- 본문 길이가 권장 범위(3500~4500자) 안에 있음
- 톤이 한쪽으로 통일됨
- AI 사용 안내 박스가 글 제목 바로 아래에 있음
- `corrections` 배열과 본문 정정 안내가 일관됨
- 글 하단 "이 글에 대하여" 섹션이 존재함

검사 결과 문제가 발견되면 Claude Code는 위치와 내용만 보고하고 자동으로 고치지 않습니다. 사용자가 직접 수정한 다음 검사를 다시 돌립니다.

## 단계 5 — 네이버 블로그 발행 (컴파일 작업)

발행 작업은 GitHub의 소스 파일을 네이버 블로그의 결과물로 "컴파일"하는 작업입니다. 사용자가 직접 컴파일러 역할을 합니다.

발행 직전에 frontmatter 블록을 제외한 본문만 복사할 준비를 합니다. frontmatter는 GitHub에만 있고 네이버 블로그에는 옮기지 않습니다.

다음 요소들은 네이버 블로그에도 그대로 옮겨야 합니다.

- 글 제목 바로 아래의 AI 사용 안내 박스
- 정정 안내 박스가 있다면 그것 (AI 안내 박스보다 위에)
- 본문의 참조 블록 (원문 인용 포함)
- 글 하단 "정정 이력" 섹션 (있는 경우)
- 글 하단 "이 글에 대하여" 섹션

본문 안에 있는 마크다운 표현 중 일부는 네이버 블로그 에디터에서 다르게 렌더링될 수 있습니다. 인용 블록(`>`)이나 표 같은 요소는 옮긴 후 시각적으로 한 번 확인합니다.

발행이 완료되면 네이버 블로그 글의 URL을 복사해 두고, Claude Code에서 다음 작업을 합니다.

1. frontmatter의 `naver_url`에 URL을 적습니다.
2. `date_published`에 오늘 날짜를 적습니다.
3. `status`를 `published`로 바꿉니다.
4. 파일을 `drafts/`에서 `posts/`로 옮깁니다.
5. Linear 이슈의 상태를 "Done"으로 옮기고 이슈 코멘트에 네이버 블로그 URL을 적습니다.

## 단계 6 — 발행 후 모니터링

글을 발행하고 나면 네이버 블로그의 댓글이나 트래픽을 일정 기간 모니터링합니다. 독자의 반응에서 새로운 글 주제를 얻을 수도 있고, 글의 어떤 부분이 잘 작동하는지에 대한 단서를 얻을 수도 있습니다.

특별히 주의해서 봐야 할 것은 사실 오류에 대한 지적입니다. 댓글에서 "이 부분은 OO이 맞다"는 식의 지적이 들어오면 일차 자료를 다시 확인하고, 지적이 맞다면 단계 7로 갑니다.

## 단계 7 — 정정이 필요할 때

Claude Code에서 `/add-correction`을 실행합니다. 명령이 정정의 무게(factual / minor)를 묻습니다.

**factual 정정 (사실, 출처, 인용 오류)** 흐름:

1. frontmatter `corrections` 배열에 항목 추가 (`correction_type: factual`)
2. GitHub Issue 생성 (자동 또는 수동)
3. Issue URL을 `github_issue` 필드에 추가
4. 본문 상단에 정정 안내 박스 노출 (Issue 링크 포함)
5. 본문 안 해당 위치에 정정 표시
6. 본문 하단 "정정 이력" 섹션에 자세한 설명 추가 (Issue 없이도 충분히 이해 가능하게)

**minor 정정 (오타, 표기)** 흐름은 같지만 GitHub Issue 단계를 건너뜁니다.

GitHub에서 정정이 완료되면 **즉시 네이버 블로그에도 동일하게 반영합니다.** GitHub는 소스 파일이므로 먼저 고치지만, 네이버는 결과물이므로 가능한 빨리 동기화해야 합니다. 두 곳이 어긋난 상태로 오래 두지 않습니다.

원본 본문은 어떤 경우에도 지우지 않습니다.

## 단계 8 — 후속 글로 이어질 때

시간이 지나 같은 주제로 후속 글을 쓰게 되는 경우, 새 글로 시작합니다(`/new-post`). 단계 1의 이전 글 매칭 단계에서 이전 글을 강한 후보로 발견하고, 이전 글의 내용을 갱신하는 성격이라면 새 글의 `supersedes`에 이전 글의 slug를 적고, 이전 글의 `superseded_by`에 새 글의 slug를 적습니다.

이전 글을 지우거나 archived 상태로 옮기지 않습니다. 두 글이 모두 살아 있어야 독자가 사용자의 입장이 시간에 따라 어떻게 진화했는지 추적할 수 있습니다.

## 정기 점검

분기에 한 번 정도, 모든 발행된 글의 `volatility`를 확인하고 점검 대상을 선정합니다. `volatility: high`인 글 중 발행 후 6개월이 지난 글이 우선 점검 대상입니다.

Claude Code에게 "high 볼라틸리티 글 중 최근 6개월 이상 점검 안 한 글을 알려줘"라고 시킬 수 있습니다. frontmatter의 데이터를 통해 자동으로 추출됩니다. 정보가 여전히 정확하면 그대로 두고, 변경이 있으면 단계 7의 정정 절차를 시작합니다.

## Linear ↔ GitHub 연동

이 저장소의 이슈는 `yohanpark-dev-docs` 팀(이슈 키 `YDOC`)의 "네이버 블로그 글" 프로젝트에서 관리하며, Linear와 GitHub 저장소가 양방향으로 연결되어 있습니다. 실무에서 쓰는 방법은 다음과 같습니다.

- 커밋 메시지나 PR 제목·본문에 이슈 키(`YDOC-3` 등)를 적으면 해당 Linear 이슈에 그 커밋·PR이 자동으로 연결됩니다. 이슈 키는 글 frontmatter의 `linear_issue_id`와 같은 값이므로, 초안 작업을 커밋할 때 이 키를 메시지에 넣어두면 Linear에서 글의 진행 상황을 한자리에서 볼 수 있습니다.
- 이슈를 "Done"으로 옮기는 시점은 깃 머지가 아니라 **네이버 블로그 발행**입니다(단계 5). 따라서 PR에 `Closes YDOC-N` 같은 자동 종료 문구는 쓰지 않습니다. 발행이 끝난 뒤 사람이 직접 Done으로 옮깁니다.

## 도구별 역할 정리

이 워크플로우 안에서 각 도구의 역할을 한눈에 정리하면 다음과 같습니다.

사용자는 주제 발굴, 본문 작성, 평가와 결론, 최종 결정, 그리고 네이버 블로그로의 컴파일을 모두 직접 합니다. Claude Code는 일정 제안, 파일 생성, 자료 검색과 정리, frontmatter 자동 채우기, 발행 전 검사, 정정 자동화, 이전 글 매칭 후보 제시를 담당합니다. Linear는 일정 관리와 상태 추적을 담당하며, GitHub과 양방향으로 연결됩니다. GitHub Issue는 사실 오류 정정의 트래킹을 담당하고, 네이버 블로그는 최종 결과물 출판지입니다.
